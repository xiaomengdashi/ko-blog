## 1. 背景与核心原理

视频探讨了如何从零开始构建一个类似 **Claude Code** 的 AI 编程智能体（Agent）。通过开源项目 **Galaxy** 的 5 个版本（v0 到 v4），渐进式地揭露了 AI 编程智能体的底层原理。

### 核心疑问
- **定位与修改**：AI 如何知道修改哪个文件？如何定位问题？
- **连续操作**：普通聊天机器人只能回复文字，为何 AI Agent 能连续执行多文件改代码、跑命令等实际操作？
- **理解能力**：AI 是真正理解代码和依赖，还是模式匹配？为何有时会"遗忘"或"跑偏"？
- **成本问题**：为何任务消耗几十万 Token？如何优化效率？

### 核心原理：循环与工具
AI Agent 的核心原理非常简单，没有复杂的神经网络魔法，本质是一个 **While 循环**：
1.  **模型思考**：决定调用什么工具。
2.  **代码执行**：环境调用工具并返回结果。
3.  **反馈循环**：结果返回给模型，模型继续思考。
4.  **终止条件**：直到模型认为任务完成，循环结束。

**四大关键概念组合**：
1.  **工具（Tools）**：赋予行动力（如 `Read`, `Write`, `Bash`），将"会说"变为"会做"。
2.  **循环（Loop）**：赋予自主性，从一问一答变为自主完成任务。
3.  **上下文（Context）**：赋予记忆，工具调用结果追加到消息历史，形成工作记忆。
4.  **提示词（Prompt）**：赋予方向，系统提示词定义角色和工具描述，指导模型拆解任务。

### 常见痛点与原因
- **上下文遗忘**：上下文窗口有限，早期内容被截断或压缩。
- **幻觉（Hallucination）**：模型基于概率匹配，可能编造不存在的 API 或文件路径。
- **任务跑偏**：缺乏明确的任务追踪，容易在长操作中迷失方向。
- **Token 消耗巨大**：每次循环都发送完整消息历史，读取大文件会永久占用上下文。
- **速度慢**：每次工具调用都是一次完整的网络请求。
- **信任与安全**：将键盘交给概率模型存在风险，需权限控制和沙箱检查。

## 2. V0 版本：Bash 即一切

V0 版本仅用约 **50 行代码** 和一个工具实现了完整的 Agent 能力。

### 核心洞察：Unix 哲学
- **一切皆文件，一切皆管道**：**Bash** 是 Unix 世界的入口。
- **单一工具**：只需要 `Bash` 一个工具，就能实现读文件、写文件、探索、执行甚至创建子代理。
- **递归调用**：通过 Bash 调用自身脚本，即可实现子代理（Sub-Agent），无需额外的 Agent 框架。

### 代码实现逻辑
1.  **初始化**：创建助手对象，传入用户提示词和历史记录。
2.  **While 循环**：
    -   调用模型，传入历史记录。
    -   判断返回结果：
        -   若是文本：直接返回结果，结束任务。
        -   若是工具调用：提取 Bash 命令，在本地执行。
    -   将执行结果追加到历史记录。
    -   进入下一轮循环。
3.  **提示词约束**：
    -   优先使用工具，少说多做。
    -   复杂任务可通过 Bash 调用自身创建子代理，隔离上下文。

### 演示效果
-   输入"创建一个 hello.txt 文件"，模型返回 Bash 工具调用，执行后返回结果。
-   输入"总结整个项目"，模型递归调用自身（子代理）进行探索，主代理保持上下文整洁。

## 3. V1 版本：模型即代理

V1 版本扩展为约 **200 行代码**，引入了 **4 个核心工具**，覆盖 90% 的场景。

### 核心转变：从聊天机器人到自主代理
-   **传统聊天**：用户问 -> 模型答（文本）。
-   **Agent 系统**：用户问 -> 模型思考 -> 调用工具 -> 执行结果 -> 模型再回复。
-   **决策者**：模型决定何时调用工具、调用顺序及何时停止。

### 四大核心工具
1.  **Bash**：执行命令（需注意危险命令拦截）。
2.  **ReadFile**：读取文件内容（可限制行数）。
3.  **WriteFile**：写入文件内容。
4.  **EditFile**：编辑文件（替换特定内容）。

### 代码结构
-   **While 循环**：持续调用模型，直到无工具调用。
-   **工具路由**：根据模型返回的工具名称（如 `ReadFile`），执行对应的 Python 函数。
-   **透明性**：每一步工具调用和结果都在对话上下文中清晰可见。
-   **可扩展性**：新增工具只需添加函数和描述，无需改动核心架构。

## 4. V2 版本：结构化规划与约束

V2 版本约 **300 行代码**，增加了一个 **Todo 工具**，解决了模型"遗忘"和"跑偏"的问题。

### 核心问题：计划在大模型脑中易丢失
-   随着对话变长，早期计划会被挤出上下文窗口。
-   模型在多步操作后容易忘记初始目标。

### 解决方案：Todo 管理
-   **结构化列表**：计划写在 Todo 列表中，每次调用都能看到完整清单。
-   **实时状态**：标记任务为 `pending`, `in_progress`, `done`。
-   **三条硬规则**：
    1.  **限制数量**：最多规划 20 条待办，防止过度规划。
    2.  **单线程执行**：每次只能完成一个任务，避免分心。
    3.  **强制更新**：每次执行计划必须更新状态；若超过 10 轮未更新，强制提醒模型更新 Todo。

### 设计哲学
-   **约束产生智能**：限制时间（或步骤）才能抓住重点。
-   **自我强化反馈**：模型调用 Todo 工具 -> 看到完整列表 -> 执行当前任务 -> 更新状态。
-   **效果**：用户可一目了然看到进度，模型不易走偏。

## 5. V3 版本：子代理与上下文隔离

V3 版本约 **500 行代码**，实现了 **子代理（Sub-Agent）机制**，解决上下文污染和 Token 消耗问题。

### 核心问题：单体 Agent 的局限性
-   **上下文限制**：探索大量文件后，历史信息过长，导致后续任务失去焦点。
-   **Token 浪费**：无关的探索细节一直占用上下文。

### 解决方案：分而治之
-   **任务工具（Task Tool）**：主代理通过 `Task` 工具创建子代理。
-   **上下文隔离**：子代理运行在独立进程中，拥有独立的上下文，主代理只接收最终结果摘要。
-   **角色专业化**：定义三种专用代理类型：
    1.  **Explorer**：仅负责探索和分析代码（只读工具）。
    2.  **Coder**：负责具体编码和修复 Bug（全工具）。
    3.  **Planner**：负责规划策略（只规划，不改代码）。

### 工作流程
1.  主代理分析需求。
2.  调用 `Task` 工具创建 `Planner` 子代理制定计划。
3.  创建 `Coder` 子代理执行具体编码。
4.  子代理完成后返回摘要，主代理汇总结果。

## 6. V4 版本：Skills 技能机制

V4 版本引入了 **Skills 机制**，实现了从"训练 AI"到"教育 AI"的转变。

### 核心理念：知识外化
-   **传统方式**：知识锁在模型参数中，需昂贵训练和微调。
-   **新范式**：知识外化为可编辑的文档（如 `skills.md`），零成本、可版本控制、可共享。
-   **区别**：
    -   **工具（Tools）**：定义模型**能做什么**（能力边界，如读写文件）。
    -   **技能（Skills）**：定义模型**知道怎么做**（领域知识，如 PDF 处理规范）。

### Skills 三层结构（按需加载）
1.  **元数据层**：名称、描述、触发条件（始终加载，节省 Token）。
2.  **内容层（skills.md）**：详细步骤、规则、示例（匹配后加载）。
3.  **资源层**：关联的配置文件或参考代码（按需访问）。

### 实现逻辑
-   **Skills Loader**：启动时扫描目录，解析 Skills 元数据。
-   **Skills 工具**：模型在遇到特定领域任务时，调用 `Skills` 工具加载对应知识包。
-   **动态注入**：加载后的技能内容作为提示词注入上下文，指导模型行为。

## 7. 课程总结与核心洞察

回顾从 V0 到 V4 的构建旅程，AI Agent 系统可归纳为四个正交层：

1.  **工具层（Tools）**：定义能力边界。从 V0 的单一 Bash 到 V1 的多工具，再到 V3 的角色专属工具。
2.  **规划层（Todo）**：将隐式计划转化为显式列表，通过硬约束（数量、单线程）引导模型聚焦。
3.  **架构层（Sub-Agents）**：通过子代理实现上下文隔离和任务分发，主代理作为协调者（Coordinator）。
4.  **知识层（Skills）**：知识不再依赖参数，而是外化为文档，支持增量更新和社区共享。

### 六大核心结论
1.  **Agent = While 循环**：核心模式始终是"思考 - 调用 - 反馈"。
2.  **工具定义能力**：给模型什么工具，它就有什么能力。
3.  **约束产生智能**：明确的规矩（如 Todo 限制）让模型更靠谱。
4.  **分而治之**：复杂任务通过子代理隔离上下文，各司其职。
5.  **知识≠参数**：经验可外化为 Skills 文件，复用且低成本。
6.  **渐进式构建**：不要一步到位，每次增加一点功能并验证，逐步理解系统全貌。

## AI 总结

本视频通过从零构建 5 个版本的 Mini Claude Code，深入浅出地拆解了 AI 编程智能体的核心原理。

-   **本质**：AI Agent 并非黑盒魔法，而是 **While 循环 + 工具调用 + 上下文记忆** 的组合。
-   **演进路线**：
    -   **V0**：验证核心循环，Bash 即一切。
    -   **V1**：细化能力，引入专用工具（读/写/改）。
    -   **V2**：增强稳定性，引入 Todo 列表进行结构化规划和约束。
    -   **V3**：解决复杂性，引入子代理实现上下文隔离和角色分工。
    -   **V4**：增强知识，引入 Skills 机制将领域知识外化为文档，实现低成本的知识注入。
-   **核心价值**：理解了这些原理，开发者可以摆脱对复杂框架的依赖，根据需求灵活构建高效、可控的 AI 应用。约束和外化知识是提升 Agent 可靠性的关键。
