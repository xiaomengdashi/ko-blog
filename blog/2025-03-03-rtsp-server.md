## 项目背景与目标

本项目旨在开发一个基于 **C++** 的 **RTSP** 流媒体服务器，支持摄像头实时流和本地文件流的传输。

1. **开发初衷**
   - 解决摄像头资源独占问题：通常一个摄像头只能被一个应用独占。
   - 实现流分发：通过 RTSP 服务器将一路视频流分发给多个客户端（CS 架构或 BS 架构）。
   - 统一接口：上层应用只需接收 RTSP 流，无需关心底层硬件细节。

2. **技术栈选型**
   - **编程语言**：C++11/14（兼容较老系统）。
   - **摄像头捕获**：Windows 平台使用 **Media Foundation API**。
   - **RTSP 协议**：集成 **Live555** 开源框架（避免重复造轮子）。
   - **Web 管理界面**：使用 C++ Web 框架（转录中提及 OTBP/Crowd 类似框架）实现状态监控。
   - **开发模式**：全项目由 **AI** 辅助生成，耗时约两天。

## AI 驱动的开发策略

在 AI 辅助编程中，开发者需转变角色，从"代码编写者"变为"技术专家/管理者"。

1. **人机协作模式**
   - **角色定位**：开发者是专家，AI 是执行力强的技术骨干。
   - **任务拆解**：不能直接丢给 AI 一个大问题（如"写一个 RTSP 服务器"），需将任务拆解为小模块。
   - **迭代调整**：查看 AI 输出结果 -> 调整指令 -> 再次生成。

2. **效率提升**
   - 利用 AI 快速集成复杂框架（如 Live555），自动处理编译和依赖。
   - 重点放在架构设计和逻辑校验上，而非底层语法。

## AI Agent 核心架构原理

视频深入探讨了支撑 AI 编程工具（如 Claude Code、Cursor）背后的核心逻辑，即 **Agent Loop**。

1. **核心循环 (The Loop)**
   - 本质是一个 `while True` 死循环。
   - **流程**：用户指令 -> 模型思考 -> 调用工具 -> 获取结果 -> 反馈给模型 -> 继续思考。
   - **决策权**：模型决定调用什么工具、以什么顺序、何时停止。

2. **神奇效果的来源**
   - **工具赋予行动力**：普通 LLM 只能输出文字，赋予 Bash/File 工具后可操作系统。
   - **循环赋予自主性**：通过循环，模型可以从"一问一答"变为"自主完成任务"。
   - **上下文赋予记忆**：历史消息作为工作记忆，让模型知道之前做了什么。
   - **提示词赋予方向**：System Prompt 规定模型的行为准则（如"少说多做"）。

3. **用户痛点与解决方案**
   - **上下文遗忘**：引入 **Todo 列表** 将计划显性化，防止长对话中迷失方向。
   - **AI 幻觉**：限制工具调用范围，通过概率匹配减少虚构 API。
   - **Token 消耗**：优化上下文管理，精简历史记录。
   - **安全风险**：设置危险命令拦截（如 `rm -rf`），权限控制。

4. **最小化工具集**
   - 覆盖 90% 场景的核心工具仅需 4 个：
     1. **Bash**：执行 Shell 命令。
     2. **Read_File**：读取文件内容。
     3. **Write_File**：写入/创建文件。
     4. **Edit_File**：编辑/替换文件内容。

## RTSP 服务器技术实现

1. **协议栈实现**
   - **RTSP**：应用层协议，负责控制（播放、暂停、 teardown）。
   - **RTP**：传输层协议，负责实际音视频数据传输。
   - **RTCP**：配合 RTP，进行传输质量控制。
   - **SDP**：会话描述协议，协商媒体类型（如 H.264, MJPEG）。

2. **关键模块**
   - **RTSP Manager**：管理流的生命周期。
   - **Camera Capture**：基于 Media Foundation 捕获摄像头数据。
   - **File Source**：读取本地 H.264 文件并模拟流传输。
   - **Web Control**：提供 HTTP 接口，用于启动/停止服务、查看会话状态。

3. **资源竞争解决 (Mutex)**
   - **问题**：频繁打开/关闭摄像头会导致硬件资源状态不稳定（No Signal）。
   - **方案**：采用 **互斥锁 (Mutex)** 和 **单例模式**。
   - **逻辑**：确保摄像头初始化只执行一次，避免重复操作导致的资源冲突。

## 进阶功能：结构化规划与 Todo

为了增强 AI Agent 处理复杂任务的能力，引入了 **结构化规划 (Structured Planning)** 机制。

1. **V2 版本改进**
   - 增加 **TodoWrite** 工具。
   - 解决长任务中模型"忘记目标"的问题。

2. **Todo 管理规则**
   - **显性化**：计划不再只在模型"脑中"，而是写入结构化列表。
   - **状态追踪**：每个任务包含 `content` (内容), `status` (状态), `activeForm` (当前动作)。
   - **约束机制**：
     - 最多 20 条待办（防止过载）。
     - 同时只能进行 1 个任务（防止分心）。
     - 必须填写完整状态（强制规范）。

3. **软约束与反馈**
   - **鼓励而非强制**：在 System Prompt 中建议模型使用 Todo，而非强制。
   - **自我强化**：写入 Todo -> 看到 Todo -> 更新 Todo，形成闭环反馈。
   - **提醒机制**：如果超过一定轮次未使用 Todo，系统会自动注入提醒消息。

## 项目演示与测试

1. **服务部署**
   - 编译生成 `.exe` 文件，双击运行。
   - 后台自动运行，监听端口（如 RTSP 554, Web 5050）。
   - 启动时自动创建默认流（文件流 + 摄像头流）以便测试。

2. **客户端播放**
   - 支持 **VLC Player**、**PotPlayer**、**FFplay** 等标准播放器。
   - **测试命令**：`ffplay rtsp://localhost:554/camera1`。
   - **多路复用**：同一摄像头流可同时被多个播放器打开，无资源冲突。

3. **性能表现**
   - **延迟**：存在轻微延迟，主要源于客户端软解码（FFplay 默认配置），非服务器端问题。
   - **稳定性**：解决了摄像头重复初始化问题后，推流稳定。
   - **管理界面**：可通过 Web 页面实时查看在线会话、流量状态，并手动启停流。

## AI 总结

本项目是一个典型的 **AI 辅助软件工程** 案例。它不仅仅展示了一个 C++ RTSP 服务器的实现，更核心的是展示了如何利用 **AI Agent** 的思维模式来构建复杂系统。

1. **技术价值**：实现了低延迟、多路复用的 RTSP 流媒体服务，解决了硬件资源独占痛点。
2. **方法论价值**：验证了"专家 +AI 骨干"的协作模式。通过任务拆解、工具封装（Bash/File/Todo）和循环反馈机制，AI 能够独立完成从架构设计到代码落地的全过程。
3. **未来展望**：随着 AI 模型能力的提升，开发者将更多专注于 **规则制定** 和 **结果验收**，而将具体的实现细节交给 AI Agent 完成。结构化规划（Todo）和权限控制是构建可靠 AI 系统的关键。
